# Scenario 42: JPA Auditing (@CreatedDate)

Demonstrates how to automatically track when a record was created and last modified without manual code.

## Concept
Compliance and debugging often require knowing "Who changed what and when?". JPA Auditing automates the "When" part using lifecycle listeners.

## Implementation Details
1. Enabled auditing via `@EnableJpaAuditing` in a config class.
2. Added `@CreatedDate` and `@LastModifiedDate` to the `Customer` entity.

### Entity Snippet:
```java
@CreatedDate
@Column(updatable = false)
private LocalDateTime createdDate;

@LastModifiedDate
private LocalDateTime lastModifiedDate;
```

## Verification Results
- **Action**: Call `/api/scenario42/create`.
- **Result**: The response shows a `createdDate` timestamp even though we never called `setCreatedDate()` in our code.
- **Update**: If you update the user later, the `lastModifiedDate` will automatically refresh.

## Interview Theory: Auditing
- **The "Who"**: You can also use `@CreatedBy` and `@LastModifiedBy`. Combined with Spring Security, these can automatically store the `username` of the person who made the change.
- **Standardization**: It's best practice to put these fields in a `@MappedSuperclass` (e.g., `BaseEntity`) so every table in your database has consistent auditing columns.
- **Precision**: Mention that these timestamps are generated by the Application (JPA), not the Database default values.
